This page lists common migration issues and safe rollback practices.

== Known issues to watch

Common categories during Java runtime migrations:

* Plugin initialization failures due to outdated dependencies
* Illegal-access or module encapsulation warnings
* Agent startup failures from mismatched JVM versions
* Native package or image mismatch on specific architectures

== Troubleshooting workflow

. Confirm controller Java version in System Information.
. Confirm agent Java versions on all active nodes.
. Identify first failing plugin or component from startup logs.
. Update or temporarily disable incompatible plugins.
. Re-run a reduced validation job set, then full workload.

== Example symptoms and actions

|===
|Symptom |Likely cause |Action

|Agent fails to reconnect after controller upgrade
|Agent JVM not updated to Java 25
|Upgrade agent JVM, reconnect agent, verify remoting logs

|Plugin fails during startup
|Plugin version not compatible with current core/Java
|Upgrade plugin to latest compatible release, retest

|Repeated reflective access warnings
|Legacy library behavior
|Check for plugin/library updates and track warnings over time
|===

== Rollback procedure

If production validation fails, rollback quickly:

. Stop Jenkins controller.
. Restore previous Java runtime (for example Java 21 or Java 17, as previously used).
. Restore `JENKINS_HOME` backup when data consistency is in doubt.
. Restart Jenkins and verify controller health.
. Reconnect agents on the same Java version as controller.
. Re-run smoke jobs to confirm service recovery.

[IMPORTANT]
====
Avoid partial rollback states where controller and agents run different major Java versions.
====

== Recovery recommendations

* Keep startup/service configuration snapshots before migration.
* Keep tested rollback runbooks with owner contacts.
* Define a maximum rollback decision time in the maintenance plan.
* Capture all migration logs for post-incident review.
